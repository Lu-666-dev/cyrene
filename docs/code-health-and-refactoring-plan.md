# Cyrene 代码健康评估与重构计划

更新日期：2026-06-29

## 1. 结论

当前仓库的顶层结构方向正确：Tauri 负责系统能力，Model 页面负责渲染与交互，`packages/content` 负责内容协议和校验。项目还不是不可维护的“屎山”，但运行时代码已经出现明显的集中化和双实现问题。

如果继续直接向 `pet-main.ts` 和 `App.tsx` 添加功能，预计再增加两到三个中等复杂度功能后，修改范围、回归风险和调试成本会明显失控。因此建议先进行一次小范围、保持行为不变的拆分，再继续开发新功能。

当前唯一产品里程碑是完成 Electron 到 Tauri 的迁移，补齐桌面宠物功能并保证开发环境稳定运行。当前不制作安装器，也不进行正式发布。喂养、宠物状态、动态插件和插件 SDK 已从项目范围移除；结构优化只处理会阻碍功能完善或造成明确运行时缺陷的问题，不扩展成独立的平台化工程。

## 2. 当前结构

```text
apps/desktop/src-tauri
  Tauri 窗口、托盘、鼠标穿透、光标采样和原生命令

apps/model-lab/src/pet-main.ts
  桌面宠物加载、渲染、布局、交互、命中检测和动作播放

apps/model-lab/src/ui/App.tsx
  模型实验室 UI、模型预览、动作编辑和动作流播放

packages/content
  内容包、Live2D 模型和动作协议的解析与校验

packages/shared-types
  跨包共享的数据类型

pets
  模型资源的唯一源目录
```

值得保留的设计：

- 原生能力集中在 Tauri 层，没有散落到前端业务代码。
- `packages/content` 将外部 JSON 视为不可信输入，并提供解析和引用校验。
- TypeScript 开启了 `strict`、`noUncheckedIndexedAccess` 和 `exactOptionalPropertyTypes`。
- `pets/` 是资源源目录，Vite 的 public 目录由脚本生成。
- 喂养、宠物状态、插件系统和 SDK 已从当前范围移除，避免了不必要的产品与抽象成本。

## 3. 主要风险

### 3.1 桌面端和实验室存在两套协议及播放逻辑

风险等级：高

`App.tsx` 使用 `@cyrene/content` 解析内容包、模型和动作，而 `pet-main.ts` 重新声明了 `ContentPackDescriptor`、`Live2DActionMapping` 等接口，并通过类型断言直接使用 JSON。

影响：

- 内容协议修改时，需要同步修改两套实现。
- 实验室校验通过不代表桌面端行为正确。
- 同一个动作可能在预览和桌面端表现不同。
- 无效内容在桌面端可能退化为静默失败。

处理原则：桌面端和实验室必须共享同一套内容加载和动作执行核心。

### 3.2 两个上帝文件承担过多职责

风险等级：高

`pet-main.ts` 超过 1300 行，同时管理：

- 内容加载
- PixiJS 和 Live2D 生命周期
- 模型布局与缩放
- 鼠标穿透和窗口形状
- 拖动与滚轮交互
- 像素命中检测
- 交互区域匹配
- 动作播放和串联
- 调试覆盖层

`App.tsx` 同样超过 1300 行，并集中维护大量 React 状态、模型生命周期、动作编辑、动作流播放和完整页面结构。

影响：任何功能都可能修改同一文件，代码冲突和隐式副作用会持续增加，纯逻辑也难以单元测试。

### 3.3 动作完成等待存在定时器泄漏

风险等级：中

`waitForMotionFinish` 使用 50ms 定时轮询检查动作是否取消。动作正常完成或 3 秒超时后，Promise 虽然已经结束，但轮询不会停止，直到后续动作改变序号。

影响：长时间运行或频繁触发动作后，会累积无效定时器。

处理原则：统一使用可取消的播放会话，并在完成、取消和超时三个出口清理所有监听器与定时器。

### 3.4 自动验证没有覆盖真实桌面运行路径

风险等级：中

当前脚本存在以下缺口：

- `check:all` 只执行 TypeScript 和 `cargo check`，不执行 Vite 生产构建。
- `smoke` 不检查 Rust 桌面端。
- 动作流校验脚本没有进入统一检查链。
- 没有桌面窗口启动和资源加载的最小集成测试。

本次桌面窗口问题能够通过现有检查，说明检查结果还不能代表功能完整或真实运行稳定。

### 3.5 桌面桥接层丢弃异步错误

风险等级：中

`tauri-desktop.ts` 中多个 `invoke` 被 `void` 丢弃，调用方无法等待或捕获失败。托盘图标、鼠标穿透或拖动状态同步失败时，容易形成未处理的 Promise rejection。

处理原则：桥接层方法统一返回 `Promise<void>`，由控制器决定记录、重试或降级。

### 3.6 资源同步不是严格镜像

风险等级：低

`sync-pets.mjs` 只复制新增或变化文件，不删除目标目录中已经从源目录移除的文件。旧资源可能残留在生产构建中。

处理原则：同步前生成源文件清单，并安全删除目标目录内不在清单中的文件。

## 4. 目标模块边界

第一阶段不新增 workspace package，先在 `apps/model-lab/src` 内建立清晰边界：

```text
apps/model-lab/src
  runtime/
    content-loader.ts       内容获取、解析和校验
    live2d-model.ts         模型创建、销毁和基础变换
    action-player.ts        动作、串联、取消和播放会话
    action-flow.ts          动作并行与队列编译

  pet/
    pet-controller.ts       桌面宠物生命周期编排
    model-layout.ts         定位、缩放和视口约束
    interaction.ts          点击、拖动和滚轮输入
    hit-test.ts             像素及区域命中检测
    debug-overlay.ts        命中边界和调试信息

  desktop/
    bridge.ts               Tauri 命令和事件适配
    window-shape.ts         窗口形状同步和错误降级

  ui/
    App.tsx                 页面组合，不承载播放实现
    hooks/
      use-model-runtime.ts
      use-action-buffer.ts
      use-action-playback.ts
    components/
      model-preview.tsx
      action-library.tsx
      action-buffer.tsx
      saved-states.tsx
```

只有当其他应用也需要复用这些模块时，再将稳定的 `runtime` 提升为 workspace package。不要为了目录整洁提前创建复杂框架。

## 5. 分阶段实施计划

### 阶段 A：先建立安全网

目标：重构过程中能够证明行为没有退化。

任务：

1. 将 `verify:action-flow` 纳入统一检查命令。
2. 增加 Vite 生产构建检查。
3. 为内容解析器、视口约束和动作取消增加单元测试。
4. 增加桌面端启动后窗口存在、资源入口可加载的最小 smoke test。
5. 保留当前桌面交互作为手工回归清单：显示、拖动、缩放、点击区域、鼠标穿透和托盘操作。

### 阶段 B：统一内容加载

目标：删除桌面端协议双写。

任务：

1. 提取 `runtime/content-loader.ts`。
2. 统一调用 `parseContentPackManifest`、`parseLive2DModelSettingsCatalog`、`parseLive2DActionMap` 和交互区域校验。
3. `pet-main.ts` 与 `App.tsx` 使用同一个加载结果类型。
4. 移除 `pet-main.ts` 内重复声明的内容协议接口和类型断言。

### 阶段 C：统一动作播放器

目标：桌面端和实验室使用相同动作语义。

任务：

1. 提取可测试的 `ActionPlayer`。
2. 使用播放会话 ID 或 `AbortController` 统一处理取消。
3. 在完成、取消、超时和销毁时清理监听器及定时器。
4. 将 before/after、表达式、动作优先级和动作流逻辑收敛到同一实现。

### 阶段 D：拆分页面和交互控制器

目标：入口文件只负责组装。

任务：

1. 将桌面布局、输入、命中检测和调试覆盖层分别提取。
2. 将实验室状态拆入 hooks，将页面区域拆为组件。
3. 保持组件属性以业务数据和命令为主，避免传递 PixiJS 内部对象。
4. 将 Tauri bridge 改为完整的异步接口并处理错误。

### 阶段 E：资源与开发环境治理

目标：让开发资源和验证结果可预测。

任务：

1. 将资源同步改为安全镜像。
2. 提供 `clean` 和开发缓存说明，避免 Rust `target` 目录造成误判。
3. 将开发检查、编译验证和桌面 smoke test 整合为单一验证命令。

## 6. 新功能开发约束

在重构完成前，新功能遵循以下规则：

- 不再向 `pet-main.ts` 添加新的独立子系统。
- 不在 `App.tsx` 内新增可以独立测试的业务算法。
- 内容 JSON 必须经过 `packages/content` 解析，不允许新增 `response.json() as SomeType`。
- 原生系统能力必须通过 Tauri bridge 暴露，前端不能直接感知平台实现。
- 所有长期运行的定时器、事件监听器和动画帧必须有明确销毁路径。
- 新增动作语义必须同时覆盖桌面端和实验室，并有自动测试。
- 先提取稳定职责，不引入通用事件总线、插件系统或依赖注入框架。

## 7. 验收标准

本轮结构优化完成时，应满足：

- `pet-main.ts` 只保留启动和模块组装，目标少于 250 行。
- `App.tsx` 只保留页面组合和少量页面级状态，目标少于 400 行。
- 桌面端和实验室共享内容加载及动作播放核心。
- 桌面入口不再重复声明内容包和动作协议。
- 动作播放连续运行后没有持续增长的定时器或监听器。
- Tauri bridge 的失败可被调用方观察和处理。
- 删除源模型文件后，public 目录不会残留旧资源。
- 一个统一命令能够完成类型检查、单元测试、内容校验、前端生产构建和 Rust 检查。
- 桌面宠物的显示、拖动、缩放、点击、穿透和托盘功能保持现有行为。

## 8. 优先级建议

必须在新增复杂功能前完成：阶段 A、阶段 B、定时器泄漏修复。

基础迁移完成后按功能完善需要逐步完成：阶段 C、阶段 D；不为拆分而拆分。

不属于当前里程碑：安装器、商店发布、完整发布版集成测试和 workspace package 提取。
